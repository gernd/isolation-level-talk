Isolation

  • concurrent transactions should be logically serialized
  • DBMS is responsible for preventing failures caused by concurrent transactions
  • several isolation level exist
  • determine what transactions can see from other concurrent transactions
  • characterized by bad things that are guaranteed not to happen
  • tradeof between data integrity and performance
  • defined by SQL standard

Dirty Read

One transaction reads uncommitted data from another transaction

Dirty Read

  • First transaction
  • Inserts user into user table
  • Inserts entry in user_group table so that user is in Admin group

Dirty Read

  • Second transaction
  • Gets all users and their associated groups

Dirty Read

  • T1: insert user
  • T2: get all users with groups
  • T1: insert entry in user_groups table
  • effect: user is in no group

Non-repeatable Read

A row is retrieved twice within a transaction and the values of the row differ between reads

Non-repeatable Read

  • First transaction: Selects users with their last login date
  • Second transaction: Updates login date for for one user

Non-repeatable Read

  • T1: Select users
  • T2: Update login date for one user
  • T1: Select users
  • effect: login date from second select differs for one user

Phantom Read

Result of query changes during transaction because of updated/deleted rows

Phantom Read

  • First transaction
  • Get sum of all prices of an shopping_item table
  • Get number of all items in shopping_item table
  • Application code: Calculate average price (sum of all prices / number of items contained in shopping_item table)

Phantom Read

  • Second transaction
  • Delete item from shopping_item_table

Phantom Read

  • T1: Get sum of all prices of an shopping_item table
  • T2: Delete item from shopping_item_table
  • T1: Get number of all items in shopping_item table
  • T1: Application code: Calculate average price (sum of all prices / number of items contained in shopping_item table)
  • effect: calculated average price is wrong

MVCC

MVCC

  • multiversion concurrency control
  • provide efficient multiuser concurrent access for DB systems
  • each transaction sees a snapshot of the data (database version)
  • no read locks required (as each transaction has its own version for reading)
  • more costs for managing several DB versions for each transaction

Optimistic Concurrency Control

  • version information is saved for each row
  • transaction remembers version number for each row it reads or updates
  • when committing, it checks whether the affected rows have been changes by another transaction
  • handling of potential conflict needed (e.g. rollback/retry)
    • TODO example for isolation level x

Pessimistic Concurrency Control

  • DB item (e.g. row) is exclusively locked by one transaction
  • Granularity: Rows, Tables, Pages
  • Read/write locks
  • deadlocks possible, increases waiting time for other transactions
  • Once a transaction has modified/read a DB item, lock is kept until transaction has committed

  • Deadlock can be detected or transaction timeout occurs
  • One of the transactions ("victim") is rolled back
